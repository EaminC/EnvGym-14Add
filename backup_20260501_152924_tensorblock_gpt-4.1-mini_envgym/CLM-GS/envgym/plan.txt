=== ADJUSTED ENVIRONMENT SETUP PLAN (for CPU-only Linux x86_64 system) ===

---

**Summary:**  
Your hardware environment is Linux x86_64 without an NVIDIA GPU. Consequently, CUDA, GPU drivers, and GPU-accelerated PyTorch builds are not applicable. The plan below is adjusted to fully support CPU-only operation, leveraging CPU training/offload modes, and ensuring compatibility with your hardware. Storage and memory requirements remain important and unchanged. This plan assumes no GPU acceleration and instructs necessary changes accordingly.

---

1. DOWNLOADS NEEDED (ADJUSTED FOR CPU-ONLY):

- Git (latest stable version) for cloning the repository  
- Conda (Anaconda or Miniconda) with Python 3.10 environment support  
- **No NVIDIA GPU driver or CUDA Toolkit installation required** (remove all CUDA 12.4 and related dependencies)  
- PyTorch 2.6.0 **CPU-only version** (install CPU-only PyTorch build)  
- Additional Python packages: torchvision 0.21.0, torchaudio 2.6.0 (CPU-only builds if applicable)  
- Python packages (via pip): tqdm, plyfile, psutil, numba, opencv-python, scipy, matplotlib, pandas, imageio, imageio-ffmpeg, requests, tabulate  
- Build tools for compiling submodules (e.g., gcc/g++, setuptools) compatible with your Linux OS and x86_64 architecture  
- Optional: Black code formatter (pip install black)  
- Adequate local disk storage:  
  - Minimum ~500GB free disk space recommended for MatrixCity BigCity dataset decoded images (as before)  
  - At least 50GB free disk for models/outputs  
  - Fast local disk strongly recommended especially for large datasets (avoid network filesystems)  
- System memory:  
  - Minimum 32GB RAM recommended for regular experiments  
  - For large-scale datasets (BigCity 102M), 100GB+ CPU RAM recommended for pinned memory allocations  
  - Mip360 and Rubble 4K datasets’ CPU RAM requirements remain unchanged  
- Optional: bash shell for running example scripts  
- Notes on offload strategies:  
  - `--clm_offload` **can still be used** in CPU-only mode but CUDA kernels in `clm_kernels` are CUDA-based and require GPU presence; thus, **`--clm_offload` cannot be used on CPU-only systems** because it depends on CUDA and GPU drivers  
  - `--naive_offload` uses CPU pinned memory but also depends on CUDA GPU for data transfer and training; thus, **`--naive_offload` is not applicable without GPU**  
  - **Only `--no_offload` training strategy is feasible and supported on your CPU-only system** (full training on CPU without GPU offloading)  
  - `--no_offload` uses `gsplat` rendering kernels which are native CPU-compatible (compiled submodules support CPU)  
- Version compatibility requirements:  
  - Python 3.10  
  - PyTorch 2.6.0 CPU-only build  
  - Compatible build tools (gcc/g++, make, python-dev headers) for compiling submodules  
- Environment variables related to CUDA allocator config (`PYTORCH_CUDA_ALLOC_CONF`) are **not applicable** and should be omitted
